# 前言 | Project Seed

2020年8月中，我收到訊息詢問我要不要一起挑戰第12屆IT邦幫忙鐵人賽。

當時我剛結束台東綠島五天四夜之旅回到台北，感受著皮膚被曬到通紅的微微刺痛，想到今年春夏也算是在海邊玩得盡興，既然已經達成玩到曬黑的目標，是時候也該回歸窩在電腦前的宅宅生活了，便決定來嘗試看看寫文。

第一次嘗試，最擔心的就是到中後期腦袋空空掰不滿30篇，所以就直接拉了個30列表格，能填滿表格的主題就採用，順便訂好系列文目錄大綱。

最後確定的主題是"Side Project開發紀錄"，為了湊滿30篇把專案流程前期的企畫和設計階段都排排進去。

想著這樣跑過一遍較正式的專案流程，應該能對職場上與各部門溝通合作有所幫助，也能讓最後成品更接近產品而不是Demo，或許是個還不錯的點子。



然而在賽前先試寫幾篇來了解自己產文速度的過程中，慢慢發覺自己寫文的方式和大鋼似乎並不適合鐵人賽......

其中一個原因，是主題中的Side Project並不是只為了鐵人賽而生的專案，而是已經上線運作累積了使用者回饋而打算做大改版的專案。因為太怕想不到足夠內容可寫，就直接把整個專案都當作鐵人賽的素材，而這樣做的後果就是試寫的文章裡有很多對專案重要但對鐵人賽是多餘的內容。

像是功能規劃的部份，專案要開發的功能可能要幾十個，但鐵人賽只有30天，大多數功能根本不會在鐵人賽系列後面的文章中出現。

最大的問題還是完成一篇文的時間太長，根本不可能達到鐵人賽要求的一天一篇文。



當自己終於確認目前做法不可行的時候，已經離鐵人賽開賽的最後期限沒幾天，後來找了已經完賽過的前輩請教討論和自我檢討後，最終還是決定放棄今年的鐵人賽。

不過這半個月寫文嘗試，覺得這主題對Side Project的開發還是有蠻有用處的，所以想繼續寫下去，畢竟自己還是蠻好奇一個人如果也依照專案流程來開發的話，究竟Side Project的品質會不會有所提升呢?

就來嘗試看看吧。
