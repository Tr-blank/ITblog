# 想想專案流程 | Project Seed

在鐵人賽試寫專案企劃書時，就有發現要一次將專案規劃到完整，尤其是其中有涵蓋到不熟悉的領域或技術的專案，要真的估出準確的時程和成本似乎不太現實。

就算把專案切分成數個功能來讓估時更具體，也心知肚明這種規劃與真實的開發情況差距甚遠(PM與RD在會議室裡爭執的畫面都浮現出來了)



既然決定這次Side Project除了開發以外也要重視專案流程的部分，那麼就算只是從RD的角度出發，也該試著找到真正可以流暢執行的規劃方式，而不是照班以往在職場上已經經歷過的煎熬流程。

雖然最後專案都能成功上線，但真的是打重心底的希望能有讓溝通更順暢、減少需求變更、少點臨時加班的做法阿~

想通這點後，便與幾位職場前輩聊聊各自的經驗，決定棄賽用更充裕的時間去思考整理。



聽前輩的建議去看了些敏捷開發的資料後，覺得裏頭有些理念方式可以來嘗試看看。

雖然對敏捷開發還是一知半解，不過邊摸索邊學習總比一直只在看理論來的更快理解，就直接來實作看看吧~



以我目前的理解，一開始是使用User Stories來找出專案中的核心流程，以使用者流程為單位來做短期的規劃開發。每完成一段短期的流程開發，都能檢視一遍目前專案進度的使用者回饋好壞，能較即時的調整篩選尚未執行需求的重要程度，減少重工的機率。透過完成一個個短期流程開發，讓專案保持可用狀態同時推進進度，讓專案越來越完善。

另外也決定保留部分之前寫的企畫書內容，雖然在實現方式有了更動，但主要目標還是不變的，我認為在開發的過程中，三不五時回顧一下專案的目標是蠻重要的一件事。

相信在一次次推進專案進度的過程中，現在是未知數的部分也會越來越明朗吧!
